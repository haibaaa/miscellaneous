// 01Overview & Problem Definition

The Heavy Hitters Problem:
Given a stream of n elements from a universe U, find all elements that appear more than n/k times, using sub-linear memory. The parameter k controls the frequency threshold.

Motivation

Real-world data streams are unbounded — storing every element is infeasible. Yet many applications need to know which items dominate the stream: network routers detecting traffic floods, databases identifying hot query patterns, recommendation systems finding trending items. The Misra-Gries algorithm (1982) solves this with O(k) space.

Formal Definition:
Let σ = ⟨a₁, a₂, …, aₙ⟩ be a data stream. For any threshold φ ∈ (0,1), element e is a φ-heavy hitter if its frequency f(e) ≥ φ·n. Setting k = 1/φ, we want all elements with frequency > n/k.

Intuition

1
Maintain a small "candidate" table
Keep at most k−1 (element, count) pairs. Elements not in the table are implicitly at count 0.
2
Increment on match
If the incoming element already has a counter, increment it. Simple O(1) hash lookup.
3
Insert new element if table has room
If the table has fewer than k−1 entries, add the element with count 1.
4
Decrement-and-evict when full
If the table is full and the element is new: decrement ALL existing counters by 1 and remove any that hit zero. This is the key step — it eliminates groups of k distinct elements at once.
5
Report survivors
After processing the full stream, elements remaining in the table are the heavy hitter candidates.

Historical Note

Misra and Gries published "Finding Repeated Elements" in Science of Computer Programming, 1982. The same algorithm was independently discovered by Boyer & Moore (1980) for the majority-element problem (k=2), by Demaine et al. as the "Frequent" algorithm (2002), and by Karp, Shenker & Papadimitriou in a different formulation. The k=2 case — finding the majority element — is often taught first.

// 02Algorithm Specification

Pseudocode

/**
 * Misra-Gries Heavy Hitters
 *
 * Input:  Stream σ of n elements, parameter k ≥ 2
 * Output: Set of heavy hitter candidates
 *         (guaranteed: all elements with freq > n/k are included)
 * Space:  O(k) — at most k-1 counters
 */

INIT(k):
  counters ← empty HashMap   // {element → count}

PROCESS(element e):
  if e ∈ counters:
    counters[e] ← counters[e] + 1       // increment

  else if |counters| < k - 1:
    counters[e] ← 1                      // insert new entry

  else:
    // Table full, new element → decrement all
    for each (item, cnt) in counters:
      counters[item] ← cnt - 1
    counters.remove_zeros()              // evict zeroed entries

OUTPUT():
  return keys(counters)                 // candidate set

// Optional: second pass to compute exact counts for candidates
VERIFY(candidates, stream):
  exact ← {e: 0 for e in candidates}
  for each element e in stream:
    if e ∈ exact: exact[e] += 1
  return {e: c for e,c in exact if c > n/k}

Implementation (Python)

class MisraGries:
    def __init__(self, k: int):
        """k controls threshold: heavy hitters have freq > n/k"""
        assert k >= 2, "k must be at least 2"
        self.k = k
        self.counters: dict = {}

    def process(self, element) -> None:
        if element in self.counters:
            self.counters[element] += 1

        elif len(self.counters) < self.k - 1:
            self.counters[element] = 1

        else:
            # Decrement all, remove zeros
            to_delete = []
            for key in self.counters:
                self.counters[key] -= 1
                if self.counters[key] == 0:
                    to_delete.append(key)
            for key in to_delete:
                del self.counters[key]
            # Note: incoming element is NOT added during decrement step

    def candidates(self) -> set:
        return set(self.counters.keys())

    def estimate(self, element) -> int:
        return self.counters.get(element, 0)

Efficient Implementation Note

Batch Decrement Optimization:
Instead of physically decrementing every counter one-by-one, maintain a global offset variable. When a decrement-all event occurs, increment the offset. Store counters as (nominal_value + offset). This makes decrement O(1) instead of O(k). Only subtract the offset when reading a counter value. 
class MisraGriesOptimized:
    def __init__(self, k):
        self.k = k
        self.counters = {}   # stores raw values
        self.delta = 0      # global offset subtracted on read

    def process(self, e):
        if e in self.counters:
            self.counters[e] += 1
        elif len(self.counters) < self.k - 1:
            self.counters[e] = self.delta + 1   # store raw
        else:
            self.delta += 1                      # O(1) global decrement
            self.counters = {
                k: v for k, v in self.counters.items()
                if v > self.delta                  # still positive
            }

    def count(self, e):
        return max(0, self.counters.get(e, 0) - self.delta)

// 04Error Guarantees & Analysis

The Guarantee

For every element e, let f̂(e) be the estimated count (counter value) and f(e) be the true count. After processing n elements with parameter k:

f(e) − n/k ≤ f̂(e) ≤ f(e)

The estimate is a lower bound on the true count, with additive error at most n/k.
f(e) − ⌊n/k⌋ ≤ counters[e] ≤ f(e) where n = stream length, k = parameter. Elements not in counters have an implicit count of 0.

Consequence: No False Negatives

If f(e) > n/k, then counters[e] > 0 (element must be in the table). Therefore every true heavy hitter is always a candidate. No heavy hitters are ever missed.

False Positive Bound

The table contains at most k−1 elements. So there are at most k−1 candidates reported. True heavy hitters (frequency > n/k) number at most k−1 (since their frequencies sum ≤ n). Thus all reported candidates are either true heavy hitters or have frequency in the range (n/k − n/k, n/k] = (0, n/k].

Why Counters Can Only Undercount

Claim: f̂(e) ≤ f(e)
A counter for e is incremented exactly once per occurrence of e in the stream. It is decremented only during a decrement-all event, which happens when e is NOT the current element. So the counter can never exceed the true count. ∎

// 05Correctness Proof

Theorem (Misra-Gries, 1982)

After processing a stream of n elements with parameter k, every element with true frequency > n/k appears in the counter table. Moreover, for every element in the table with counter value c̃, its true frequency f satisfies:

f ≥ c̃   and   f ≤ c̃ + n/k

Proof of No False Negatives

Key Lemma — Decrement Count Bound
Let D = total number of decrement-all events during the processing of n elements. Each decrement-all event accounts for exactly k elements: the k−1 counters that are decremented, plus the 1 new element that triggered the event. Since each element is consumed at most once, total elements processed ≥ k · D. Therefore D ≤ n/k.
Proof: Every heavy hitter survives
Let e be an element with f(e) > n/k. Counter for e is incremented f(e) times and decremented at most D ≤ n/k times. So the final counter value c̃(e) ≥ f(e) − D ≥ f(e) − n/k > 0. Since c̃(e) > 0, element e is in the table when OUTPUT() is called. ∎
Proof: Counter does not exceed true count
Counter for e is incremented only when e appears in the stream. It is decremented only during events triggered by a different element. Therefore the counter can be incremented at most f(e) times. Hence c̃(e) ≤ f(e). ∎
Proof: At most k−1 elements in table
Each decrement-all event removes at least one element (the one whose counter hits zero from above 0, or possibly several). After each such event, the table has at most k−2 entries (since all were decremented including those that evicted). Before a decrement event, the table must have had exactly k−1 entries (otherwise we would have inserted). So the table size is bounded by k−1 at all times. ∎

Space Complexity Proof

The table holds at most k−1 entries. Each entry stores an element (universe label) and a counter (integer ≤ n). With a hash map, this requires O(k) memory words. ∎

Amortized Time Analysis

Total work across all n elements
Each element either: (A) increments one counter in O(1), (B) inserts in O(1), or (C) triggers a decrement-all. Case C costs O(k) but also consumes k elements (k−1 counters + 1 triggering element). Total cost of all Case C events ≤ (n/k) · O(k) = O(n). Adding Cases A and B: total cost = O(n). Per-element amortized = O(1). ∎

Tight Example

The bound n/k error is tight. Consider the stream:

[a, b, c, a, b, c, a, b, c, …]   (k=3, groups of 3 distinct elements)

With k=3, we keep k−1=2 counters. The 3rd element always triggers a decrement-all. If this repeats n/3 times, the surviving element has counter 0 and true count n/3. Error = n/3 = n/k exactly.

// 06Extensions & Variants

1. Merge / Parallel Processing

Two Misra-Gries summaries can be merged to produce a valid summary of the union stream. This enables distributed/parallel computation. 
MERGE(A: MisraGries, B: MisraGries) → MisraGries:
  // Step 1: Combine by summing shared counters
  combined = {}
  for (e, c) in A.counters:
    combined[e] = combined.get(e, 0) + c
  for (e, c) in B.counters:
    combined[e] = combined.get(e, 0) + c

  // Step 2: Reduce to k-1 entries by finding (k+1)-th largest
  if |combined| <= k - 1:
    return combined                    // already small enough

  sorted_vals = sort(combined.values(), descending=True)
  threshold = sorted_vals[k - 1]       // (k-1)-th largest value

  return {e: c - threshold
          for (e, c) in combined.items()
          if c - threshold > 0}

Merge Error: The merged summary has error bound n_A/k + n_B/k = (n_A + n_B)/k, as expected for the combined stream of length n_A + n_B.

2. Weighted / Frequency Streams

If each element arrives with a weight w_i (e.g., packet sizes in bytes), extend Misra-Gries by adding w_i to the counter instead of 1, and decrement by min(existing_counts) during a full event. The error bound becomes W/k where W = total weight.

3. Space-Saving Algorithm (Metwally et al., 2005)

A refinement where, when the table is full and a new element arrives, the new element replaces the element with the smallest counter and inherits that counter value (instead of decrementing all). This gives tighter estimates and wastes no space.

// Space-Saving variant of PROCESS(e):
if e ∈ counters:
  counters[e] += 1
elif |counters| < k - 1:
  counters[e] = 1
else:
  min_key = argmin(counters.values())
  min_val = counters[min_key]
  del counters[min_key]
  counters[e] = min_val + 1   // inherits +1

4. Frequent Directions (Matrix Analog)

The "Frequent Directions" algorithm (Ghashami et al., 2016) is the matrix analog of Misra-Gries. Instead of decrementing scalar counters, it applies SVD and shrinks singular values — yielding a low-rank sketch with additive spectral error.

5. Count-Min Sketch (Alternative)

Count-Min Sketch uses hash functions and a 2D array to answer point queries in O(1) with multiplicative error, but requires randomness. Misra-Gries is deterministic and exact (within additive error) but limited to the top-k candidates.

6. Sliding Window Variant

For time-decaying streams, elements are weighted by a decay factor λ^t. Modified Misra-Gries periodically scales all counters by λ. Heavy hitters are those with weighted frequency above the threshold.

7. Second Pass Verification

A common two-pass pipeline: Pass 1 → Misra-Gries to get candidates. Pass 2 → scan stream once more, counting only candidates exactly. Then filter by f(e) > n/k. This gives exact answers using only O(k) memory between passes.

// 07Edge Cases & Corner Cases

k = 2 (Majority Element / Boyer-Moore)

Keeps only 1 counter. The algorithm reduces to Boyer-Moore voting. The sole candidate (if it survives) is the majority element — only one exists with freq > n/2. If no majority exists, the single survivor may be a false positive.

a b a b c → [c:1] → no majority

No Heavy Hitters Exist

Uniform stream: all n/k elements appear exactly k times. Every decrement event evicts all entries. The table may end up empty or with survivors that are false positives. The guarantee still holds: any survivor has count ≤ n/k (not heavy). No incorrect heavy hitter is reported.

Output: ∅ or false positives (to be verified)

All Elements Identical

Stream of n copies of 'a'. The table immediately gets entry {a:1} and is never full (k−1 ≥ 1). Counter increments to n. No evictions occur. 'a' is correctly identified as the only candidate.

counters = {a: n}

Exactly k Heavy Hitters

k elements each appear n/k times. Their frequencies sum to n, so they use all occurrences. Misra-Gries can keep at most k−1 counters, so one heavy hitter may be evicted if the algorithm sees elements in an adversarial order. Each true heavy hitter has freq = n/k (not strictly greater), so the guarantee doesn't apply — all could be evicted!

Single-Element Stream (n = 1)

One element arrives, table is empty, so it's inserted with count 1. Candidate set = {that element}. Heavy hitter threshold = 1/k < 1, so technically any element appearing once with n=1 is heavy. Correct behavior.

Adversarial Input (Maximum Evictions)

Input: k groups of k distinct elements, cycled. Every k-th element triggers decrement-all. Surviving elements have their counts maximally underestimated. Error = n/k (tight bound). The algorithm is still correct but estimates are least useful here.

a b c | a b c | a b c | … (k=3)

k = 1 (Degenerate)

Threshold would be > n/1 = n, which no element can satisfy. The algorithm keeps 0 counters (k−1 = 0). Every element triggers a no-op. No candidates are ever reported. Correctly: no element has frequency > n. Guard: assert k ≥ 2 in implementation.

Very Large Universe

Elements are 64-bit hashes, domain size |U| = 2⁶⁴. Misra-Gries is unaffected — it only tracks the elements it has actually seen. Memory is O(k) regardless of universe size. The hash map naturally handles sparse, large-domain inputs.

Decrement Evicts Multiple Elements

During a decrement-all, multiple elements can simultaneously hit counter 0 and be evicted. Example: all k−1 counters are at 1; new element arrives; all decrement to 0; all k−1 are evicted in one step. The incoming element is then NOT added. Table is now empty.

Stream Length Unknown

Misra-Gries works without knowing n in advance. The output is the candidate set. The error bound n/k can only be computed post-hoc once n is known (or estimated). For online threshold queries, maintain a running count of n alongside the algorithm.

Merging Summaries of Unequal Size

When merging two Misra-Gries summaries for streams of lengths n₁ and n₂, the combined error bound is n₁/k + n₂/k = (n₁+n₂)/k. The merge procedure itself must use the same k parameter for the result to be valid. Using different k values is undefined.

Counter Overflow

For very long streams (n > INT_MAX), use 64-bit or arbitrary-precision counters. With optimized offset representation, counters store raw values which grow at most as fast as n. For n = 10¹² elements, use 64-bit integers (max ~1.8 × 10¹⁸).
