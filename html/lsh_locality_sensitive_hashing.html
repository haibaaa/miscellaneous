Background

The Nearest-Neighbor Problem

Given a set P of n points in ℝ^d, find the point p ∈ P closest to a query q. The naïve scan is O(dn) — prohibitive when n or d are large. Exact solutions suffer from the curse of dimensionality: all known approaches need either exponential space or exponential time.

Curse of Dimensionality: Voronoi diagrams need O(n^⌈d/2⌉) space. k-d trees run in O(d log n) on average but O(dn) worst case. No near-linear space + sub-linear time solution is known for exact nearest-neighbor.

Exact Approaches

Naïve scanO(dn) query, O(dn) space
Voronoi (2D)O(log n) query, O(n) space
Voronoi (d-D)O(log n) query, O(n^⌈d/2⌉) space
k-d treeO(d log n) avg, O(dn) worst

Section 01

PLEB → ANN Reduction

The (r₁, r₂)-PLEB (Point Location in Equal Balls) is an intermediary problem: given query q, return a point within r₂ if one exists within r₁, or null if none exists within r₂. The "undefined" zone (point within r₂ but not r₁) can always be checked explicitly.

ANN via PLEB — Index Constructionδ = √ε
1.Set δ = √ε
2.Build (r, δr)-PLEB for each r ∈ {D_min/δ, D_min, δ·D_min, δ²·D_min, …, D_max}
3.Total instances: k ≤ 1 + log_δ(D_max/D_min)
4.Total space: O(k · S(n))
ANN Query via Binary Search on PLEBO(T(n) · log k)
1.Find smallest r* such that (r*, δr*)-PLEB returns a point p*
2.Guaranteed: dist(p*, q) ≤ δ·r* (returned point is within r₂)
3.Guaranteed: no point within r*/δ of q (else smaller PLEB would have fired)
4.Therefore: r*/δ < OPT ≤ dist(p*,q) ≤ δ·r* → p* is δ² = ε approximate
5.Binary search exploits monotonicity: O(log k) PLEB queries

Section 02

Hyperplane LSH — Cosine Similarity

Points are hashed by which side of a random hyperplane they fall on. Each hyperplane has a normal vector drawn from N(0,1)^d. The k-bit hash code captures which of k hyperplanes each point is "above" or "below". Similar points (small angle) are more likely to agree on all bits.

h(p) = sign(p · Z^T) ← 1 if dot-product ≥ 0, else 0 Pr[h(p) = h(q)] = 1 − θ/π ← where θ = angle between p and q For k hyperplanes (one hash function g ∈ G): Pr[g(p) = g(q)] = (1 − θ/π)^k = α^k α = 1 − θ/π → 1 as θ → 0 (similar points) β = 1 − θ/π → 0 as θ → π (dissimilar points) False-negative rate = (1 − α^k)^ℓ ← over ℓ sets of hyperplanes Expected bad points = n · (1−(1−β^k)^ℓ) ← via union bound
Hyperplane LSH — Hash Code Computationk hyperplanes, ℓ sets
1.Select k × ℓ random hyperplanes. Each hyperplane: d-dim normal vector from N(0,1)^d.
2.For each point p and each set j ∈ [ℓ]: compute k-bit hash gⱼ(p) = [sign(p·Z₀ᵀ), …, sign(p·Zₖ₋₁ᵀ)]
3.Store p in bucket gⱼ(p) for each j. Space: O(dn·ℓ + kn·ℓ)
4.Query q: compute ℓ hash codes, collect points from ℓ buckets, check distances explicitly.
5.Avg bucket size: n/2^k. Choosing k = log₂ n gives O(d·ℓ·log n) query time.

Section 03

LSH Family — Formal Theory

A hash family H is (r₁, r₂, α, β)-locally sensitive if near points collide with high probability (≥ α) and far points collide with low probability (≤ β). The key parameter κ = ln α / ln β governs the space-time tradeoff.

LSH Family H = {h : U → B} is (r₁, r₂, α, β)-sensitive if: dist(p,q) ≤ r₁ → Pr[h(p)=h(q)] ≥ α dist(p,q) > r₂ → Pr[h(p)=h(q)] ≤ β Compound hash family G = {g : U → B^k}: Each g = (h₀, h₁, …, hₖ₋₁), hᵢ ∈ H independently Setting k = −ln(n)/ln(β) = −log_β(n): β^k = 1/n → E[|far points in bucket|] ≤ 1 (Markov → Pr ≤ 1/2) Setting ℓ = n^κ where κ = ln α / ln β: α^k = n^{−κ} → Pr[near point in some bucket] ≥ 1−1/e ≥ 1/2 Overall success probability ≥ 1/4. Amplify with c = Θ(log n) copies → 1 − (3/4)^c.

Section 04

MinHash — Jaccard Similarity

MinHash estimates the Jaccard similarity JS(A,B) = |A∩B|/|A∪B| between sets. The key insight: a random permutation π of rows in the characteristic matrix produces minHash(A) = minHash(B) with probability exactly JS(A,B).

JS(A, B) = |A ∩ B| / |A ∪ B| ← similarity of two sets dist(A,B) = 1 − JS(A,B) ← complementary distance Key property (MinHash + Jaccard): Pr[minHash_π(A) = minHash_π(B)] = JS(A,B) Why? Among rows where at least one of A,B has a 1: x = rows where BOTH have 1 (= |A∩B|) y = rows where exactly ONE has 1 → Pr[first-1-row is same for both] = x/(x+y) = JS(A,B) ✓ ℓ permutations → estimate JS_min(A,B) = (# matching minHashes) / ℓ E[JS_min(A,B)] = JS(A,B) (unbiased!) Pr[|JS_min − JS| > λ] ≤ 2e^{−ℓλ²/2} (Hoeffding bound)
MinHash Computation (Practical)ℓ hash functions
1.Choose ℓ independent hash functions hᵣ : Σ → [0, σ−1]
2.Initialize minHash[r] = ∞ for all r
3.For each word w in document A: for each r, update minHash[r] = min(minHash[r], hᵣ(w))
4.Output ⟨minHash[0], …, minHash[ℓ−1]⟩ as the signature
5.Estimate JS: JS_min(A,B) = (# positions r where minHash_r(A) = minHash_r(B)) / ℓ

Section 05

Banding Technique

Given ℓ MinHash signatures, comparing all pairs is O(n²). Banding converts this into a probabilistic filter: divide the ℓ-length signature into b bands of r rows (ℓ = b·r). Two documents become candidate pairs if they match in at least one band. This creates an S-curve threshold effect.

Pr[match in one band] = s^r ← s = JS(A,B), all r rows agree Pr[no match in any band] = (1 − s^r)^b Pr[candidate pair] = P(s) = 1 − (1 − s^r)^b ← the S-curve! Threshold (inflection point): t = (1/b)^{1/r} = (1/b)^{b/ℓ} s >> t → P(s) → 1 (high-similarity pairs detected) s << t → P(s) → 0 (low-similarity pairs ignored) Increasing b: fewer rows/band → lower threshold → more matches → fewer FN, more FP Decreasing b: more rows/band → higher threshold → fewer matches → fewer FP, more FN
