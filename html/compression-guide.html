Chapter 01Shannon's Entropy

Entropy is the theoretical minimum average number of bits needed to represent each character in a message. It is the benchmark every compression scheme tries to reach.

H = −Σ (nᵢ/n) · log₂(nᵢ/n)

Where nᵢ is the frequency of character i and n is the total message length. Higher entropy = less compressible (more random). Lower entropy = more structure = more compressible.

Key Insight
No lossless compression can shorten every possible input — by the Pigeonhole Principle, at least two strings would map to the same code, making decoding impossible. Compression exploits structure in data.

Chapter 02Huffman Coding

Huffman coding is optimal for symbol-by-symbol prefix coding. It assigns shorter codes to more frequent characters by building a binary tree greedily — always merging the two lowest-frequency nodes first.

Prefix-Free Property
No codeword is a prefix of another. This guarantees unambiguous decoding — just scan bits left-to-right and output whenever you hit a leaf of the tree.

Block Huffman

Standard Huffman assigns 1 bit minimum per character. For highly skewed distributions this is suboptimal. Block Huffman treats sequences of k characters as a single meta-character, allowing fractional bits per original character.

Caveat
The alphabet grows as σᵏ — exponentially. For a binary alphabet with block size 2, we get 4 meta-characters. With block size 10, we get 2¹⁰=1024 meta-characters. The mapping table overhead can eliminate gains for small strings.

Chapter 03Shannon-Fano Encoding

Shannon-Fano assigns code lengths of ⌈-log₂(pᵢ)⌉ to each character, then assigns actual codes in lexicographic order while maintaining the prefix-free property.

ℓᵢ = ⌈-log₂(pᵢ)⌉ = ⌈-log₂(nᵢ/n)⌉
Shannon-Fano vs Huffman
Shannon-Fano is not optimal — it may assign the same code length to characters with different frequencies, wasting bits. Huffman fixes this by building the tree bottom-up from actual frequencies.

Chapter 04Arithmetic Coding

Arithmetic coding encodes an entire message as a single fraction in [0, 1). The interval is progressively narrowed with each character. The final interval's lower bound (or a value within it) is the code.

Unlike Huffman, arithmetic coding can achieve compression arbitrarily close to entropy because it is not limited to integer-length codes per character.

ℓᵢ = ℓᵢ₋₁ + δᵢ₋₁ · CumulativePr(S[i]) δᵢ = δᵢ₋₁ · Pr(S[i])

Converting to a Bit String

The final interval has size Δ. We need α = 1 + ⌈-log₂Δ⌉ bits. We use the first α bits of the binary representation of L + Δ/2, which is guaranteed to:

  • Fall within [L, L+Δ) — ensuring correctness
  • Be prefix-free — ensuring unique decodability

Chapter 05Burrows-Wheeler Transform

The BWT is a reversible permutation of the input string. It rearranges characters so that similar contexts cluster together, creating long runs that can be compressed effectively with RLE or MTF.

Key Properties
• The BWT itself has the same entropy as the original string
• It is fully reversible using the LF-Mapping
• The string must end in a unique sentinel character $ (lexicographically smallest)

LF-Mapping & Inverse Transform

LF(i) maps a position in the last column (BWT) to its corresponding position in the first column. The formula is:

LF(i) = C(BWT[i]) + rank(i) − 1

Where C(c) = number of characters lexicographically smaller than c, and rank(i) = occurrences of BWT[i] in BWT[0..i].

Chapter 06Run-Length Encoding (RLBWT)

A run is a maximal sequence of identical characters. RLE replaces runs longer than 2 characters with (character, length) pairs. After BWT, characters tend to cluster into long runs, making RLE particularly effective.

Run of length ≤ 2: keep as-is Run of length > 2: replace with char + count

Chapter 07Move-to-Front Encoding

MTF encodes each character by its position in a maintained list. After encoding, that character moves to position 0. Since BWT clusters same characters, recently used characters stay near position 0, creating a distribution heavy with small numbers — which entropy coders compress well.

Chapter 08Full Compression Pipeline (bzip2)

Real-world compressors like bzip2 chain multiple transforms. Each stage reduces structure that the next stage can exploit. The full pipeline is:

Input → BWT → MTF → (RLE of zeros) → Huffman → Compressed Output

Why the Order Matters

Compression Chain Logic
• BWT: Groups similar contexts → creates runs of same characters
• MTF: Turns runs into streams of small numbers (many zeros)
• Huffman/Arithmetic: Encodes skewed distribution efficiently

Applying MTF without BWT first yields worse results because characters don't cluster.
